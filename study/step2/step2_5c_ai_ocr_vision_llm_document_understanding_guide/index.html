    Step2-5C. AI OCR / Vision LLM 기반 문서 이해 가이드

    1. 이 문서의 목적

    Step2-5에서는 사내 문서를 대상으로 RAG를 구축한다. 사내 문서는 단순한 텍스트 파일만 존재하지 않는다. 실제 업무 문서는 PDF, PPTX, DOCX, XLSX, HWP, 이미지, 스캔 문서, 화면 캡처, 아키텍처 다이어그램, 표, 조직도, 업무 흐름도 등 다양한 형태로 존재한다.

    이 문서의 목적은 모든 문서를 무조건 텍스트 파서로 처리하는 방식이 아니라, 문서의 성격에 따라 적절한 처리 방식을 선택하는 기준을 정리하는 것이다.

    특히 다음 내용을 이해하는 것을 목표로 한다.

    • 일반 Parser 방식이 적합한 문서
    • OCR이 필요한 문서
    • AI OCR이 필요한 문서
    • Vision LLM이 필요한 문서
    • 문서 유형별 RAG 처리 전략
    • 기업형 문서 RAG에서 문서 이해 계층을 설계하는 방법

    2. 왜 모든 문서를 무조건 파싱하면 비효율적인가

    RAG를 처음 구축할 때 가장 쉽게 생각하는 방식은 모든 문서를 텍스트로 추출한 뒤 Chunk로 나누고 Vector DB에 적재하는 것이다.

    문서 수집 → 텍스트 추출 → Chunk 분할 → Embedding → Vector DB 저장 → 검색 → LLM 답변

    이 방식은 Markdown, TXT, 일반 DOCX, 일반 PDF처럼 텍스트 구조가 명확한 문서에는 잘 맞는다. 하지만 실제 사내 문서는 다음과 같은 문제가 많다.

    • PPT 문서의 핵심 내용이 도형, 화살표, 아이콘, 이미지 안에 들어있다.
    • PDF가 텍스트 PDF가 아니라 스캔 이미지 PDF이다.
    • 표 구조가 깨져서 행과 열의 관계가 사라진다.
    • 업무 흐름도, 시스템 구성도, 아키텍처 그림은 텍스트 파서로 의미를 추출하기 어렵다.
    • 화면 캡처 기반 매뉴얼은 텍스트보다 이미지 안의 UI 정보가 중요하다.
    • 엑셀 문서는 셀 값뿐 아니라 시트명, 병합 셀, 표 구조, 수식, 헤더 관계가 중요하다.
    • HWP 문서는 도구 및 라이브러리 제약이 있을 수 있다.

    따라서 모든 문서를 동일한 방식으로 파싱하면 처리 비용은 증가하지만 검색 품질은 오히려 떨어질 수 있다.

    3. 문서 처리 방식의 큰 분류

    사내 문서 RAG에서는 문서 처리 방식을 크게 네 가지로 나누어 볼 수 있다.

    구분 설명 대표 대상
    Parser 기반 추출 파일 내부의 텍스트와 구조를 직접 추출 TXT, MD, DOCX, 텍스트 PDF
    OCR 기반 추출 이미지 안의 글자를 인식하여 텍스트로 변환 스캔 PDF, 이미지 파일
    AI OCR 기반 추출 OCR 결과를 보정하고 표/문맥/레이아웃까지 해석 계약서, 신청서, 표가 많은 스캔 문서
    Vision LLM 기반 이해 이미지를 보고 의미, 구조, 관계, 설명을 생성 PPT 다이어그램, 아키텍처 그림, 화면 캡처

    중요한 점은 Parser, OCR, Vision LLM이 서로 대체 관계만은 아니라는 것이다. 실제 시스템에서는 이들을 조합해서 사용한다.

    4. Parser 기반 처리가 적합한 문서

    Parser 기반 처리는 문서 내부에 텍스트 데이터가 명확하게 들어 있는 경우에 가장 효율적이다.

    4.1 적합한 문서 예시

    • Markdown 문서
    • TXT 문서
    • 일반 DOCX 보고서
    • 텍스트 기반 PDF
    • 표 구조가 단순한 XLSX
    • HTML 문서
    • 시스템 매뉴얼 중 텍스트 위주의 문서

    4.2 장점

    • 처리 속도가 빠르다.
    • 비용이 낮다.
    • 구조화된 텍스트를 비교적 안정적으로 얻을 수 있다.
    • Embedding과 Vector DB 적재가 쉽다.

    4.3 한계

    • 이미지 안의 글자는 추출하지 못한다.
    • 도형, 화살표, 다이어그램의 의미를 이해하지 못한다.
    • PPT의 시각적 배치 의미를 놓칠 수 있다.
    • 표가 복잡하면 행과 열 관계가 깨질 수 있다.

    5. OCR이 필요한 문서

    OCR은 Optical Character Recognition의 약자로, 이미지 안에 있는 글자를 인식해서 텍스트로 변환하는 기술이다.

    5.1 OCR이 필요한 경우

    • 스캔된 PDF
    • 이미지로 저장된 계약서
    • 사진으로 촬영한 문서
    • 팩스 문서
    • 캡처 이미지 안의 텍스트
    • 텍스트 선택이 되지 않는 PDF

    5.2 OCR 처리 흐름

    이미지 또는 스캔 PDF
 → 이미지 전처리
 → OCR 수행
 → 텍스트 추출
 → 후처리
 → Chunk 분할
 → Embedding
 → Vector DB 저장

    5.3 일반 OCR의 한계

    일반 OCR은 글자를 읽는 데 초점이 있다. 하지만 사내 문서 RAG에서는 글자를 읽는 것만으로 충분하지 않은 경우가 많다.

    예를 들어 신청서나 계약서에서는 다음 정보가 중요하다.

    • 항목명과 값의 관계
    • 표의 행/열 관계
    • 체크박스 선택 여부
    • 문서 제목과 본문 영역 구분
    • 서명, 도장, 날짜 위치
    • 여러 페이지에 걸친 문맥

    일반 OCR은 이런 구조적 의미를 완벽하게 이해하지 못할 수 있다.

    6. AI OCR이 필요한 문서

    AI OCR은 단순히 글자를 인식하는 수준을 넘어, 문서의 레이아웃과 의미를 함께 이해하려는 접근이다.

    6.1 AI OCR의 역할

    AI OCR은 다음과 같은 작업에 사용된다.

    • OCR 결과의 오탈자 보정
    • 문서 영역 분리
    • 표 구조 인식
    • 항목명과 값 매핑
    • 체크박스, 도장, 서명 영역 판단
    • 문서 유형 분류
    • 핵심 필드 추출

    6.2 AI OCR이 적합한 문서

    • 신청서
    • 계약서
    • 전표
    • 증빙 문서
    • 스캔된 보고서
    • 표가 많은 PDF
    • 양식 기반 문서
    • 공문서

    6.3 예시

    일반 OCR 결과는 다음처럼 나올 수 있다.

    성명 김장관 부서 AI Lab 직책 PM 신청일 2026.06.20

    AI OCR은 이를 다음처럼 구조화할 수 있다.

    {
  "성명": "김장관",
  "부서": "AI Lab",
  "직책": "PM",
  "신청일": "2026-06-20"
}

    RAG 관점에서는 단순 텍스트보다 구조화된 데이터가 검색과 답변 품질에 훨씬 유리하다.

    7. Vision LLM이 필요한 문서

    Vision LLM은 이미지를 입력으로 받아 그 안의 내용, 구조, 관계, 의미를 설명할 수 있는 멀티모달 모델이다.

    일반 OCR이 글자를 읽는 기술이라면, Vision LLM은 이미지를 보고 설명하고 해석하는 기술에 가깝다.

    7.1 Vision LLM이 적합한 문서

    • PPT 아키텍처 다이어그램
    • 업무 흐름도
    • 시스템 구성도
    • 네트워크 구성도
    • 조직도
    • 화면 캡처 기반 매뉴얼
    • 표와 그래프가 섞인 보고서
    • 이미지 중심의 제안서
    • UI/UX 설계서

    7.2 Vision LLM이 필요한 이유

    PPT에 아래와 같은 시스템 구성도가 있다고 가정한다.

    사용자 → Web Server → API Gateway → RAG Server → Vector DB
                                      ↓
                                    LLM Server

    일반 Parser는 도형 안의 텍스트를 일부 추출할 수는 있지만, 화살표의 방향과 구성 요소 간의 관계를 안정적으로 설명하지 못한다.

    Vision LLM은 이 그림을 보고 다음과 같은 설명을 생성할 수 있다.

    이 아키텍처는 사용자가 Web Server를 통해 접속하고, 요청이 API Gateway를 거쳐 RAG Server로 전달되는 구조이다. RAG Server는 Vector DB에서 관련 문서를 검색하고, LLM Server를 호출하여 최종 답변을 생성한다.

    이렇게 생성된 설명은 RAG 검색용 문서로 활용할 수 있다.

    7.3 로컬 Vision LLM 모델 선택 기준

    Vision LLM은 반드시 이미지 입력을 지원하는 모델을 사용해야 한다. 일반 텍스트 모델인 llama3.1:8b 같은 모델은 PDF 페이지 PNG를 입력으로 받아 분석하는 용도로 사용할 수 없다.

    처음 실습할 때는 작은 모델로 흐름을 검증한 뒤, 표·차트·다이어그램 해석 품질이 부족할 때 더 큰 모델로 바꾸는 방식이 좋다.

    모델 권장 용도 실습 난이도 특징 및 주의사항
    gemma3:4b 첫 Vision LLM 실습, PPT/PDF 페이지 요약, 이미지 속 일반 텍스트 확인 낮음 현재 실습의 기본 추천 모델이다. 이미지와 텍스트를 함께 입력할 수 있어 로컬 테스트를 시작하기 좋다.
    qwen2.5vl:7b 표, 차트, 스캔 문서, 다이어그램, 구조화 JSON 추출 비교 중간 문서·도식·레이아웃 이해와 구조화 출력 비교에 적합한 후보이다. gemma3:4b보다 모델 크기와 처리 부담이 커질 수 있다.
    qwen3-vl:8b 최신 Vision 모델 비교 실험 중간~높음 사용할 Ollama 버전이 요구사항을 충족하는지 먼저 확인해야 한다. 모델이 커질수록 로컬 메모리·처리 시간이 증가한다.

    권장 시작점
    현재 로컬 RAG 실습 환경에서는 먼저 gemma3:4b로 PDF 1~3페이지를 테스트하고, 표·도식 인식 품질이 부족한 경우 qwen2.5vl:7b로 같은 페이지를 비교하는 방식을 권장한다.

    모델 설치 예시는 다음과 같다.

    # 첫 실습용 모델
ollama pull gemma3:4b

# 표·차트·도식 비교용 모델
ollama pull qwen2.5vl:7b

# 설치된 모델 목록 확인
ollama list

    모델이 실제로 Vision 기능을 지원하는지 확인하려면 Ollama API의 모델 상세 조회를 사용할 수 있다.

    curl http://localhost:11434/api/show -d '{"model":"gemma3:4b"}'

    응답의 capabilities에 아래처럼 "vision"이 포함되어 있으면 이미지 입력을 지원하는 모델이다.

    {
  "capabilities": [
    "completion",
    "vision"
  ]
}

    7.4 PDF에 Vision LLM을 적용하는 전체 흐름

    Vision LLM은 PDF 파일 자체를 바로 해석하는 것이 아니라, PDF의 각 페이지를 이미지로 렌더링한 뒤 그 이미지를 모델에 전달하는 방식으로 적용한다.

    PDF 파일
  ↓
1. pypdf로 페이지별 일반 텍스트 추출
  ↓
2. 텍스트 품질 판단
  ├─ 텍스트가 충분한 일반 문서 → Parser 결과 사용
  └─ 텍스트가 부족하거나 표/차트/도식 중심 → Vision 처리 대상
                                               ↓
3. PyMuPDF로 PDF 페이지를 PNG로 렌더링
                                               ↓
4. PNG + 프롬프트를 Ollama Vision LLM에 전달
                                               ↓
5. 이미지 속 텍스트 / 표 / 차트 / 도식 설명 생성
                                               ↓
6. Parser 원문 + Vision 결과 병합
                                               ↓
7. JSONL 저장 → Chunk → Embedding → Vector DB

    예를 들어 제안서 장표형 PDF는 pypdf가 제목·페이지 번호·일부 글자만 추출할 수 있다. 하지만 시스템 구성도 안의 서버 수, 영역별 배치, 화살표 방향, 업무 흐름은 텍스트만으로 이해하기 어렵다. 이 경우 해당 페이지를 PNG로 렌더링하고 Vision LLM으로 설명을 생성해야 한다.

    7.5.3 모델과 Ollama 서버 확인

    7.6 PDF Vision LLM 결과 예시

    Vision LLM을 사용하면 단순 Parser 결과와 시각 기반 설명을 분리해서 보관하는 것이 좋다.

    {
  "id": "architecture_proposal-pdf_vision-3",
  "text": "[PDF 원본 텍스트]\nGalaxy Store 시스템 구성도\n\n[이미지 기반 보조 텍스트]\nWeb(11), WAS(29), Oracle DB(2)\n\n[시각 요소 설명]\n아일랜드 AWS, 네덜란드 SPC, 중국 AWS 환경이 좌측·중앙·우측으로 배치되어 있다. 중앙 SPC 영역에는 Admin Site 및 Seller Portal과 Web, WAS, Oracle DB 구성이 표시되어 있으며, 양쪽 AWS 환경과 ETL/API 방식으로 연결된다.",
  "native_text": "Galaxy Store 시스템 구성도 ...",
  "vision_text": "Web(11), WAS(29), Oracle DB(2)",
  "visual_summary": "아일랜드 AWS, 네덜란드 SPC, 중국 AWS 환경이 ...",
  "chart_summary": "",
  "metadata": {
    "document_type": "pdf",
    "page_no": 3,
    "extraction_method": "hybrid",
    "extractor": "pypdf + pymupdf + ollama-vision",
    "vision_model": "gemma3:4b",
    "page_image_path": "rendered_pages/pdf/architecture_proposal/page_003.png"
  }
}

    여기서 native_text는 Parser가 직접 읽은 원문이고, vision_text와 visual_summary는 AI가 이미지에서 생성한 보조 정보다. 두 정보를 분리해 저장하면 나중에 AI가 생성한 설명만 수정하거나 재생성하기 쉽고, 원문과 생성 결과를 구분할 수 있다.

    7.7 전체 PDF 처리 시 Vision 적용 대상 판단

    모든 PDF 페이지를 무조건 Vision LLM으로 처리하면 시간이 오래 걸리고, 로컬 PC 자원 사용량도 커질 수 있다. 따라서 1차 Parser 결과를 기준으로 Vision 적용 대상을 정하는 것이 좋다.

    가장 단순한 기준은 추출된 텍스트 길이다.

    USE_VISION_LLM = True

# 이 글자 수보다 짧으면 텍스트 추출 품질이 낮다고 판단한다.
VISION_TEXT_MIN_LENGTH = 300

should_use_vision = (
    USE_VISION_LLM
    and len(native_text) < VISION_TEXT_MIN_LENGTH
)

    위 기준은 다음과 같이 동작한다.

    일반 텍스트 PDF
→ native_text 길이가 충분함
→ Parser 결과만 사용

스캔 PDF / 그림 중심 장표 / 차트·도식 중심 페이지
→ native_text가 비어 있거나 매우 짧음
→ PNG 렌더링
→ Vision LLM 호출
→ Parser + Vision 결과 병합

    다만 텍스트가 어느 정도 추출되더라도, 제안서 장표처럼 핵심 의미가 도식·화살표·배치에 있는 페이지는 Vision 처리가 필요할 수 있다. 테스트 단계에서는 특정 문서에 한해 모든 페이지를 Vision 처리해 결과를 비교해보는 것이 좋다.

    # 특정 문서의 모든 페이지를 비교 분석할 때
should_use_vision = USE_VISION_LLM

    7.8 09_extract_pdf_vision.py와의 연결

    Vision 모델 설치와 Ollama 서버 연결 확인이 끝나면, 전체 PDF 처리 파일인 09_extract_pdf_vision.py를 실행한다.

    09_extract_pdf.py

09_extract_pdf_vision.py

    권장 실행 순서는 다음과 같다.

    # 1. 텍스트 Parser 결과 확인
python 09_extract_pdf.py

# 2. 전체 PDF Vision 처리
python 09_extract_pdf_vision.py

    전체 처리 결과는 다음 경로에 저장하도록 구성한다.

    extracted_text/
├─ pdf_extracted.jsonl
└─ pdf_vision_extracted.jsonl

rendered_pages/
└─ pdf/
   └─ PDF파일명/
      ├─ page_001.png
      ├─ page_002.png
      └─ ...

    7.9 Vision LLM 프롬프트 작성 원칙

    문서 RAG 목적의 Vision 프롬프트는 단순히 “이미지를 설명해줘”라고 요청하는 것보다, 다음 항목을 분명하게 지시하는 것이 좋다.

    - 문서/슬라이드의 주제
- 주요 구성 요소
- 구성 요소 간 연결 관계
- 화살표 방향과 처리 흐름
- 표의 헤더와 행/열 관계
- 차트의 사실 기반 추세
- 읽기 어려운 텍스트는 추측하지 않기
- JSON 형태의 구조화 응답

    특히 업무 문서에서는 모델이 추측으로 내용을 보완하지 않도록 아래와 같은 문장을 프롬프트에 포함하는 것이 중요하다.

    이미지에서 확인할 수 없는 정보는 생성하지 마세요.
읽기 어려운 수치나 텍스트는 추측하지 마세요.
화살표 방향, 제목, 표 헤더, 숫자는 이미지에서 확인되는 경우에만 작성하세요.

    7.10 주요 오류와 점검 방법

    증상 가능한 원인 확인 및 조치
    Connection refused 또는 URLError Ollama 서버가 실행되지 않음 ollama list, curl http://localhost:11434/api/tags 실행
    이미지 입력 오류 텍스트 전용 모델 사용 gemma3:4b, qwen2.5vl:7b 등 Vision 모델 사용 및 capabilities 확인
    PNG는 생성되지만 분석이 매우 느림 DPI가 높거나 모델이 큼 RENDER_DPI = 120~150부터 테스트, 한 페이지씩 실행
    작은 글자 인식이 부정확함 PNG 해상도 부족 RENDER_DPI = 180~200으로 올려 비교
    표 행·열 관계가 깨짐 복잡한 표/작은 글씨 Vision 결과를 원문 Parser 결과와 함께 저장하고 중요한 표는 검토
    JSON 파싱 오류 모델이 JSON 외 텍스트를 함께 응답 format에 JSON Schema 지정, temperature = 0 유지
    텍스트가 일부 있는데 Vision이 실행되지 않음 native_text가 빈 문자열이 아님 len(native_text) < VISION_TEXT_MIN_LENGTH 기준으로 변경

    8. PPT 문서 처리 전략

    사내 문서 RAG에서 PPT는 매우 중요하다. 제안서, 보고서, 아키텍처 설명, 업무 프로세스, 교육자료가 대부분 PPT로 작성되는 경우가 많기 때문이다.

    8.1 PPT의 특징

    PPT는 일반 문서와 다르게 다음 요소가 섞여 있다.

    • 제목
    • 본문 텍스트
    • 도형
    • 화살표
    • 이미지
    • 표
    • 아이콘
    • 차트
    • 슬라이드 노트
    • 시각적 배치

    따라서 PPT는 단순 텍스트 추출만으로는 의미가 부족할 수 있다.

    8.2 PPT 처리 방식

    PPT는 보통 다음 두 가지 결과물을 함께 만든다.

    결과물 설명 활용
    텍스트 추출 결과 슬라이드 제목, 본문, 표 텍스트 기본 검색
    Vision 설명 결과 슬라이드 이미지 기반 요약 및 구조 설명 의미 검색

    8.3 권장 처리 흐름

    PPTX 파일
 → 슬라이드별 텍스트 추출
 → 슬라이드별 이미지 변환
 → Vision LLM으로 슬라이드 설명 생성
 → 텍스트 추출 결과와 Vision 설명 결과 병합
 → 슬라이드 단위 Chunk 생성
 → Metadata 부여
 → Embedding
 → Vector DB 저장

    8.4 PPT Chunk 예시

    {
  "document_type": "pptx",
  "file_name": "AI_Data_Platform_Architecture.pptx",
  "slide_no": 7,
  "title": "RAG 시스템 구성도",
  "text": "Web Server, API Gateway, RAG Server, Vector DB, LLM Server",
  "vision_summary": "사용자 요청이 Web Server와 API Gateway를 거쳐 RAG Server로 전달되고, RAG Server는 Vector DB와 LLM Server를 사용하여 답변을 생성하는 구조이다.",
  "chunk_text": "[슬라이드 7] RAG 시스템 구성도 ..."
}

    9. PDF 문서 처리 전략

    PDF는 크게 두 가지로 나눌 수 있다.

    PDF 유형 설명 처리 방식
    텍스트 PDF 문서 내부에 텍스트 정보가 있음 Parser 기반 추출
    이미지 PDF 스캔 이미지로 구성됨 OCR 또는 AI OCR 필요

    9.1 텍스트 PDF

    텍스트 선택과 복사가 가능한 PDF는 우선 Parser 기반으로 처리한다.

    PDF → 텍스트 추출 → 문단 정리 → Chunk 분할 → Embedding

    9.2 이미지 PDF

    스캔 문서처럼 텍스트 선택이 되지 않는 PDF는 페이지를 이미지로 변환한 뒤 OCR 또는 Vision LLM을 적용한다.

    PDF → 페이지 이미지 변환 → OCR / AI OCR / Vision LLM → 텍스트 및 설명 생성

    9.3 혼합 PDF

    실무에서는 텍스트와 이미지가 섞인 PDF도 많다. 이 경우 다음 방식이 좋다.

    1. Parser로 텍스트 추출 시도
2. 추출 텍스트가 충분한지 판단
3. 텍스트가 부족한 페이지는 이미지 변환
4. OCR 또는 Vision LLM 적용
5. 페이지 단위로 결과 병합

    10. XLSX 문서 처리 전략

    엑셀 문서는 단순히 셀 값을 추출하는 것만으로는 부족하다. 엑셀에서는 행과 열의 관계, 시트명, 병합 셀, 헤더, 수식, 단위가 중요하다.

    • 시트명
    • 표 제목
    • 헤더 행
    • 데이터 행
    • 병합 셀
    • 수식
    • 단위
    • 비고

    10.2 권장 방식

    엑셀은 전체 셀을 무작정 텍스트로 이어붙이기보다 표 단위로 구조화하는 것이 좋다.

    XLSX 파일

    10.3 Chunk 예시

    {
  "document_type": "xlsx",
  "file_name": "업무요건정의서.xlsx",
  "sheet_name": "전자금융",
  "table_name": "요구사항 목록",
  "row_no": 12,
  "requirement_id": "REQ-012",
  "requirement_name": "이체 한도 관리",
  "description": "사용자별 이체 한도를 관리할 수 있어야 한다."
}

    11. HWP 문서 처리 전략

    국내 기업과 공공기관에서는 HWP 문서가 여전히 많이 사용된다. HWP는 처리 환경에 따라 파싱 난이도가 다를 수 있다.

    11.1 HWP 처리 고려사항

    • HWP 파일 버전
    • HWPX 여부
    • 텍스트 추출 가능 여부
    • 표 구조 유지 가능 여부
    • 이미지 포함 여부
    • 보안 문서 여부

    11.2 권장 방식

    가능하면 HWP를 직접 처리하기보다 중간 포맷으로 변환하는 전략을 고려한다.

    HWP/HWPX
 → PDF 또는 DOCX 변환
 → Parser 또는 OCR 처리
 → Chunk 생성
 → Vector DB 적재

    단, 변환 과정에서 표나 이미지가 깨질 수 있으므로 변환 품질 검증이 필요하다.

    12. 문서 유형별 처리 판단 기준

    문서 처리 방식은 파일 확장자만으로 결정하면 안 된다. 같은 PDF라도 텍스트 PDF인지 스캔 PDF인지에 따라 처리 방식이 달라진다.

    판단 항목 질문 처리 방향
    텍스트 추출 가능 여부 텍스트 선택/복사가 가능한가? 가능하면 Parser 우선
    이미지 비중 핵심 내용이 이미지에 있는가? Vision LLM 검토
    표 구조 표의 행/열 관계가 중요한가? 구조화 추출 필요
    스캔 여부 스캔 문서인가? OCR 또는 AI OCR 필요
    다이어그램 여부 도형/화살표 관계가 중요한가? Vision LLM 필요
    비용 중요도 대량 문서인가? Parser 우선, 필요한 문서만 Vision 적용
    정확도 중요도 계약/규정/요건 문서인가? AI OCR 또는 검증 프로세스 필요

    13. 권장 아키텍처

    기업형 문서 RAG에서는 모든 문서에 고비용 Vision LLM을 적용하지 않는다. 먼저 가벼운 방식으로 처리하고, 필요한 경우에만 고도화된 처리를 적용하는 계층형 구조가 좋다.

    문서 수집
  ↓
문서 유형 판별
  ↓
1차 Parser 처리
  ↓
품질 판단
  ├─ 충분함 → Chunk 생성
  └─ 부족함 → OCR / AI OCR / Vision LLM 처리
                    ↓
              결과 병합 및 정제
                    ↓
              Metadata 생성
                    ↓
              Embedding
                    ↓
              Vector DB 저장
                    ↓
              RAG 검색 및 답변

    14. 품질 판단 기준

    1차 Parser 처리 후 바로 Vector DB에 넣지 말고, 추출 품질을 판단하는 단계가 필요하다.

    14.1 품질 판단 예시

    • 추출된 텍스트 길이가 너무 짧지 않은가?
    • 슬라이드 제목은 추출되었는가?
    • 표의 헤더가 유지되었는가?
    • 페이지 번호, 머리말, 꼬리말만 추출된 것은 아닌가?
    • 이미지 중심 문서인데 텍스트가 거의 없는 것은 아닌가?
    • 깨진 문자나 의미 없는 공백이 많지 않은가?

    14.2 품질 점수 예시

    {
  "file_name": "architecture_proposal.pdf",
  "page_no": 3,
  "parser_text_length": 120,
  "has_table": true,
  "has_image": true,
  "quality_score": 0.42,
  "next_action": "VISION_LLM"
}

    15. Metadata 설계

    AI OCR과 Vision LLM 결과를 RAG에 활용하려면 Metadata 설계가 중요하다.

    15.1 기본 Metadata

    {
  "file_name": "문서명",
  "file_path": "문서경로",
  "document_type": "pptx/pdf/docx/xlsx/hwp/image",
  "page_no": 1,
  "slide_no": 3,
  "sheet_name": "요구사항",
  "created_at": "2026-06-20",
  "source_system": "AI-Data-Platform"
}

    15.2 추출 방식 Metadata

    {
  "extraction_method": "parser | ocr | ai_ocr | vision_llm | hybrid",
  "ocr_engine": "tesseract | cloud_ocr | ai_ocr",
  "vision_model": "vision_llm_model_name",
  "quality_score": 0.87,
  "requires_review": false
}

    15.3 의미 기반 Metadata

    {
  "document_category": "architecture",
  "business_domain": "전자금융",
  "keywords": ["RAG", "Vector DB", "LLM", "Open WebUI"],
  "summary": "RAG 기반 사내 문서 검색 시스템 구성 설명",
  "contains_diagram": true,
  "contains_table": false
}

    16. Chunk 설계

    AI OCR과 Vision LLM 결과는 원문 텍스트와 분리해서 관리할 수도 있고, 하나의 Chunk로 병합할 수도 있다.

    16.1 병합형 Chunk

    [문서명] AI Data Platform Architecture
[슬라이드] 7
[제목] RAG 시스템 구성도
[원문 텍스트]
Web Server, API Gateway, RAG Server, Vector DB, LLM Server
[Vision 설명]
사용자 요청이 Web Server와 API Gateway를 거쳐 RAG Server로 전달되고...

    장점은 검색이 쉽고 구현이 단순하다는 것이다.

    16.2 분리형 Chunk

    Chunk A: Parser로 추출한 원문 텍스트
Chunk B: Vision LLM이 생성한 이미지 설명
Chunk C: AI OCR이 추출한 표 구조 데이터

    장점은 출처와 추출 방식을 명확히 구분할 수 있다는 것이다. 단, 구현은 조금 더 복잡하다.

    17. 비용과 성능 관점

    Vision LLM과 AI OCR은 강력하지만 모든 문서에 적용하면 비용과 처리 시간이 증가한다.

    17.1 권장 원칙

    • 텍스트 추출이 잘 되는 문서는 Parser로 처리한다.
    • 스캔 문서는 OCR을 적용한다.
    • 표와 양식 문서는 AI OCR을 적용한다.
    • 다이어그램과 화면 캡처는 Vision LLM을 적용한다.
    • 중요도가 낮은 대량 문서는 저비용 방식부터 적용한다.
    • 중요도가 높은 문서는 품질 검증 단계를 추가한다.

    17.2 처리 우선순위

    1순위: Parser
2순위: OCR
3순위: AI OCR
4순위: Vision LLM
5순위: 사람 검토

    비용이 낮은 방식부터 적용하고, 품질이 부족할 때 다음 단계로 넘어가는 구조가 효율적이다.

    18. 실습 관점의 단계 구성

    Step2-5C에서는 실제 구현을 다음 순서로 진행하는 것이 좋다.

    18.1 1단계: 샘플 문서 준비

    다음 유형의 샘플 문서를 준비한다.

    • 텍스트 PDF
    • 스캔 PDF
    • PPTX 아키텍처 문서
    • 화면 캡처 이미지
    • 표가 포함된 XLSX

    18.2 2단계: Parser 결과 확인

    먼저 일반 Parser로 텍스트를 추출해본다.

    확인할 내용은 다음과 같다.

    • 텍스트가 충분히 추출되는가?
    • 표 구조가 유지되는가?
    • 이미지 안의 내용이 누락되는가?
    • 다이어그램의 의미가 사라지는가?

    18.3 3단계: OCR 결과 비교

    스캔 PDF나 이미지 문서에 OCR을 적용하고 결과를 비교한다.

    확인할 내용은 다음과 같다.

    • 글자가 정상적으로 인식되는가?
    • 한글 인식 품질은 어떤가?
    • 줄바꿈과 문단 구조가 유지되는가?
    • 표의 행/열 관계가 유지되는가?

    18.4 4단계: Vision LLM 설명 생성

    PPT 슬라이드나 다이어그램 이미지를 Vision LLM에 입력하여 설명을 생성한다.

    확인할 내용은 다음과 같다.

    • 구성 요소를 제대로 식별하는가?
    • 화살표 방향과 관계를 설명하는가?
    • 업무 흐름을 자연어로 설명하는가?
    • RAG 검색에 사용할 만한 문장으로 정리되는가?

    18.5 5단계: RAG용 Chunk 생성

    Parser 결과, OCR 결과, Vision 설명을 RAG에 넣을 수 있는 Chunk 형태로 정리한다.

    20. 주의사항

    AI OCR과 Vision LLM은 강력하지만 항상 정확한 것은 아니다. 특히 업무 문서에서는 잘못된 해석이 큰 문제가 될 수 있다.

    20.1 주의할 점

    • 모델이 이미지에 없는 내용을 추측할 수 있다.
    • 작은 글자는 잘못 인식될 수 있다.
    • 표 구조가 복잡하면 행과 열을 잘못 매핑할 수 있다.
    • 업무 용어를 일반적인 의미로 오해할 수 있다.
    • 보안 문서를 외부 API로 전송하면 안 된다.
    • 개인정보와 민감정보 처리 기준이 필요하다.

    20.2 운영 기준

    • 중요 문서는 검증 단계를 둔다.
    • 원본 파일 위치를 Metadata에 반드시 남긴다.
    • AI가 생성한 설명과 원문 추출 결과를 구분한다.
    • 외부 API 사용 여부를 보안 정책에 맞게 결정한다.
    • 민감정보 마스킹 또는 비식별화 전략을 검토한다.

    21. 최종 정리

    기업형 RAG에서 문서 처리는 단순히 텍스트를 추출하는 작업이 아니다. 문서의 형식과 의미를 이해하고, 검색 가능한 지식으로 변환하는 과정이다.

    특히 사내 문서는 PPT, PDF, DOCX, XLSX, HWP, 이미지, 스캔 문서처럼 다양한 형식으로 존재한다. 따라서 모든 문서를 동일하게 처리하는 방식은 효율적이지 않다.

    권장 전략은 다음과 같다.

    Parser로 가능한 문서는 Parser로 처리한다.
글자가 이미지에 있으면 OCR을 적용한다.
양식과 표 구조가 중요하면 AI OCR을 적용한다.
다이어그램과 화면 캡처는 Vision LLM으로 설명을 생성한다.
추출 결과에는 Metadata를 부여한다.
검색 품질을 높이기 위해 Chunk 구조를 설계한다.

    Step2-5C의 핵심은 AI OCR이나 Vision LLM 자체를 사용하는 것이 아니라, 어떤 문서에 어떤 방식을 적용할지 판단하는 기준을 세우는 것이다. 이 기준이 있어야 사내 문서 RAG가 실제 업무에서 사용할 수 있는 수준으로 발전할 수 있다.
